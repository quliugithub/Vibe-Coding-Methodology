# Vibecoding 在中大型项目中突破上下文限制（v1）

**背景前言**：Vibecoding（沉浸式/直觉式 AI 辅助编程）在小型脚本或单文件项目中体验很好；但在中大型真实项目里，受限于上下文窗口、注意力衰减和“项目记忆缺失”，很容易出现“幻觉实现”或越界改动。  
这份文档给出一套可复用的 SOP：用结构化 **Evidence Pack（证据包）** 驱动模型稳定产出可落地 diff。

---

## 0) 适用范围与不适用范围

### 适用范围
- 多模块协作、存在架构边界的业务项目
- 对“只改指定文件”“不改公共接口”有硬约束的任务
- UI/交互、依赖 API、线上报错三类信息需要合并判断的任务

### 不适用范围
- 一次性脚本、Demo、探索型 PoC（可直接轻量对话）
- 需求尚未明确、验收标准无法落地的任务
- 无法提供最基本事实（如报错、接口、设计）却要求一次修复成功的任务

---

## 1) 当前 LLM 的三大硬限制

1. **上下文窗口上限 + 注意力稀释**：上下文越长，越容易忽略关键细节。  
2. **缺乏真实项目结构感知**：不知道哪些文件是核心、哪些改了会牵一发动全身。  
3. **缺少当前事实来源**：若只靠训练记忆，容易写出过期 API 或“看起来合理但跑不通”的实现。  

---

## 2) 解决思路：把“事实”分发给模型

建议把下面这些信息源作为标准输入通道（按“职责”分层，不按“是否 MCP”分层）：

- **OpenSpec / PRD / API Contract**：需求、约束、验收标准（解决需求漂移）
- **Serena（项目记忆 MCP 数据源，如 `ARCHITECTURE.md` / 规则文档）**：边界、历史决策、禁区（解决架构越界）
- **Figma（设计平台，可通过 Figma MCP 接入）**：可量化 UI 事实（解决 UI 幻觉）
- **Context7（MCP 文档检索服务，可结合官方文档）**：最新 API 正确用法（解决 API 猜测）
- **Chrome DevTools**：运行时证据（请求、DOM、报错、性能）
- **执行型模型（Codex / Claude 等）**：基于约束输出稳定 diff

---

## 3) 核心方法：Evidence Pack（证据包）

每次派发任务时，固定提供以下信息：

1. **Goal**：目标（1~3 句）
2. **Scope**：允许修改的文件白名单
3. **Constraints**：必须/禁止事项（must / must not）
4. **UI Facts**：设计状态与数值（带单位）
5. **Tech Facts**：关键 API 的最小可用片段
6. **Runtime Evidence**：运行时证据（报错、请求、性能）
7. **Output Required**：输出格式（diff / 风险 / 回归清单）

并附带硬约束：
- 仅修改 Scope 文件
- 如需改 Scope 外文件，必须先停下询问
- 不改变公共接口（除非明确授权）

---

## 4) 轻量版 vs 完整版（降低使用成本）

### 轻量版（Lite Pack）
适用于小改动、低风险任务，至少包含：
- Goal
- Scope
- Constraints
- Output Required

### 完整版（Full Pack）
适用于联调、线上问题、UI 改造等任务，包含全部 7 项。

### 选择规则（建议）
- 仅改单文件 + 无外部依赖：Lite
- 涉及 UI、API、线上报错任一项：Full
- 涉及公共接口/数据库/核心路由：Full + 人工复核

---

## 5) 工作流拆解（5 个阶段）

### 阶段 1：需求对齐（PRD → Task & AC）
**产物**：任务清单 + 可验证 AC  
**DoD（完成判据）**：
- 每条 AC 可被测试或 DevTools 观测验证
- AC 有“触发条件 + 预期结果”

**示例 AC**
- AC1：按钮 disabled 颜色为 `#E0E0E0`，hover 无变化
- AC2：超时时 Toast 文案固定为“网络超时，请重试”
- AC3：结算埋点 event 为 `checkout_click` 且 payload 含 `cart_id`

### 阶段 2：边界注入（Serena / 项目记忆 MCP → 架构约束）
**目标**：让模型不看全仓也不乱改  
**DoD（完成判据）**：
- 明确模块职责（controller/service/domain）
- 明确禁区（公共 API、表结构、核心契约）
- 明确风格（错误处理、日志、状态管理）
- 明确已知雷区（历史事故点）

建议摘要控制在 20~40 行。

**落地示例（Serena）**
- 输入示例：
  - `controller` 只做参数校验与调用 `service`
  - `service` 负责业务编排，不直接操作 UI 状态
  - 禁区：不得修改 `/api/v1/checkout` 的响应字段
- 输出示例：仅修改 `src/pages/Cart/SubmitButton.tsx` 与 `src/utils/analytics.ts`
- Prompt 示例：`请基于上述边界修复重复提交，仅改 Scope 文件，禁止改 public interfaces。`

### 阶段 3：设计量化（Figma / Figma MCP → UI Facts）
**DoD（完成判据）**：
- 覆盖状态：default/hover/active/disabled/loading/error
- 给出关键数值：padding、gap、font-size、line-height、radius、色值
- 给出精确文案（逐字、含标点）
- 给出交互规则（点击后行为、loading 禁用项、错误提示位置）

**落地示例（Figma）**
- 输入示例：
  - default：`bg #1890FF`，`radius 6px`
  - loading：`bg #D9D9D9`，文案“处理中...”，`pointer-events: none`
  - hover（非 disabled）：颜色不变化
- 输出示例：按钮在 `loading || disabled` 时统一灰态与禁点行为
- Prompt 示例：`按 UI Facts 精确实现按钮样式，不允许自行推断未给定设计值。`

### 阶段 4：依赖对齐（Context7 MCP / 官方文档 → Tech Facts）
**DoD（完成判据）**：
- 提供 10~30 行最小可用代码片段
- 标明版本或来源
- 约束“只能按该片段用法实现”

**落地示例（Context7）**
- 输入示例：`trackEvent(eventName: string, payload: Record<string, any>)`
- 输出示例：`trackEvent('checkout_click', { cart_id })`
- Prompt 示例：`只能按 Tech Facts 使用埋点 API，不要新增或猜测其他参数签名。`

### 阶段 5：执行与验证（AI Diff → DevTools 反哺）
**DoD（完成判据）**：
- 产出可读 diff（仅 Scope 文件）
- 给出回归检查清单
- 列出风险与影响面
- Bug 时能回传 Runtime Evidence 二次修复

## 6) Prompt 约束模板（推荐）

```text
你是执行者，不做重构，不做额外优化。
先输出你对 Constraints 的理解摘要（3-5 条），再执行修改。
严格只改 Scope 文件；若需改 Scope 外文件，先停止并询问。
不修改 public interfaces（除非我明确允许）。
输出：
1) Git diff
2) 回归测试清单
3) 风险点与影响面
```

> 说明：将“思考过程”替换为“约束理解摘要 + 执行计划”更稳妥，也更利于团队审阅。

---

## 7) 安全与脱敏（必须补充）

在给模型投喂证据前，默认执行以下检查：

- 去除密钥、Token、Cookie、连接串、私有证书
- 屏蔽 PII（手机号、身份证、邮箱、姓名等）
- 日志/请求体按“最小必要字段”提供
- 线上数据优先脱敏样本，不直接贴完整生产数据
- 明确“禁止输出敏感信息”的硬约束

---

## 8) 验证与回滚（防止改完不可控）

每次任务建议附带：

- **Verification Commands**：本地验证命令（如单测、lint、关键页面操作）
- **Rollback Plan**：失败后的回滚策略（回滚提交、恢复配置、兜底开关）

最小回归清单建议包含：
- 功能主路径
- 相关边界路径
- 失败/异常路径
- 性能或并发敏感路径（若涉及）

---

## 9) 防止上下文腐化（Context Decay）

每次任务完成后做两件事：

1. **反向沉淀**：基于本次 diff 更新 `ARCHITECTURE.md` / 项目规则文档（10 行内）
2. **记忆保鲜**：把新增“硬规则”纳入下一轮 Pack 模板

目标是让团队知识随迭代自动更新，而不是靠个人记忆维持。

---

## 10) 最容易翻车的 3 个点（避坑）

1. **只给截图，不给数字事实** → UI 容易被模型瞎猜  
   - 解法：把截图转成 UI Facts（状态 + 数值 + 文案）
2. **只说“看文档”，不贴最小可用片段** → 容易写出过期 API  
   - 解法：贴 10~30 行可运行片段 + 版本信息
3. **把架构文档整本塞进上下文** → 核心任务被淹没  
   - 解法：1 页以内模块化摘要，按任务精准投喂

### 11.3 Case Study：购物车按钮样式 + 防连点 + 埋点

```markdown
- **Goal**: 修改购物车提交按钮样式，增加防连点，并新增点击埋点。
- **Scope** (files allowed): `src/pages/Cart/SubmitButton.tsx`, `src/utils/analytics.ts`
- **Constraints**:
  - 必须使用 `trackEvent`，不能直接引入第三方 SDK
  - 防连点逻辑必须在组件内部，不修改全局拦截器
- **UI Facts**:
  - default: bg `#1890FF`, radius `6px`
  - disabled/loading: bg `#D9D9D9`, `pointer-events: none`, 文案“处理中...”
- **Tech Facts**:
  - `trackEvent(eventName: string, payload: Record<string, any>)`
- **Runtime Evidence**:
  - 快速双击会触发两次 `/api/v1/checkout`，返回 `409 Conflict`
- **Verification Commands**:
  - 双击结算按钮，确认仅发起 1 次请求
  - 验证 disabled/loading 样式与文案
  - 验证埋点字段完整
- **Rollback Plan**:
  - 回滚本次提交；恢复按钮到旧逻辑；关闭新增埋点开关
- **Output Required**:
  - 先给约束理解摘要，再给 diff、回归清单、风险点
```

## 12) Specs 驱动开发落地章程

目标：把 `spec` 作为唯一规格源（SSOT），Evidence Pack 作为执行载体，代码实现作为结果产物。

### 12.1 目录规范（建议）

```text
project-root/
  docs/
    specs/
      index.md
      <module>/
        <feature>.spec.md
    ai/
      EVIDENCE_PACK_TEMPLATE.md
      packs/YYYY/MM/
```

- `docs/specs/index.md`：规格索引（功能名、Owner、状态、对应 PR）
- `docs/specs/<module>/<feature>.spec.md`：功能规格正文
- `docs/ai/packs/YYYY/MM/`：按规格派生的任务证据包

### 12.1.1 `docs/specs/index.md` 写什么（建议）

`index.md` 不是写业务细节，而是“规格总入口 + 治理台账”。建议固定 4 块内容：

- **说明区**：用途、状态定义（`Draft/In Progress/Done/Archived`）、更新规则
- **规格清单表**：`Spec ID`、标题、模块、Owner、状态、Spec 路径、Pack、PR、最后更新
- **流程规则**：如 `No Spec, No Code`、契约变更必须回写 Spec
- **归档规则**：完成后多久归档、归档路径（如 `docs/specs/archive/`）

可直接使用最小模板：

```markdown
# Specs Index

## 状态定义
- Draft / In Progress / Done / Archived

## 规格清单
| Spec ID | 标题 | 模块 | Owner | 状态 | Spec 路径 | Pack | PR | Updated |
|---|---|---|---|---|---|---|---|---|
| FEAT-123 | 购物车结算按钮 | cart | Allen | Done | docs/specs/cart/checkout-button.spec.md | docs/ai/packs/2026/02/2026-02-26_FEAT-123_checkout-click-guard.md | #123 | 2026-02-27 |

## 规则
- No Spec, No Code
- 契约变更必须回写 Spec
```

### 12.2 每份 Spec 的最小字段（必须）

- **背景与目标**：为什么做、成功标准是什么
- **In Scope / Out of Scope**：改动边界与禁区
- **业务规则**：必须/禁止事项（must / must not）
- **接口契约**：请求、响应、错误码、兼容性约束
- **AC（可验证）**：每条都能被测试或 DevTools 观测
- **风险与回滚**：已知风险、失败后的恢复方案

### 12.3 标准流程（Spec → Pack → Code）

1. 先新建或更新 Spec（未落 Spec 不写代码）
2. 评审并冻结关键约束（Scope / AC / 契约）
3. 基于 Spec 生成 Evidence Pack
4. AI/人工按 Pack 实现，仅改白名单文件
5. 用 AC + Verification Commands 完成验收
6. 合并后回写 Spec 状态与变更记录

### 12.4 强制闸门（建议写入团队规范）

- **No Spec, No Code**：没有 Spec 编号或路径，不进入开发
- **PR 必须关联 Spec**：PR 描述必须包含 Spec 路径
- **契约变更必须回写 Spec**：代码改了接口但 Spec 未更新，直接驳回
- **越界修改先中止**：发现需改 Scope 外文件时，先停下并确认
- **AC 不可验证不合并**：无法验证的 AC 视为无效 AC

### 12.5 PR 描述最小模板（可直接复用）

```markdown
- Spec: `docs/specs/<module>/<feature>.spec.md`
- Evidence Pack: `docs/ai/packs/YYYY/MM/<task>.md`
- Scope Files:
  - `...`
- AC Checklist:
  - [ ] AC-1
  - [ ] AC-2
- Verification Commands:
  - `...`
- Risks/Rollback:
  - `...`
```

### 12.6 合并后回写 Spec 状态与变更记录（必须）

合并完成后，必须在对应 `*.spec.md` 中追加“状态与变更记录”小节，最少包含：

- **状态变更**：`Draft -> In Progress -> Done`（含日期与负责人）
- **AC 结果**：逐条标记 `✅/⚠️/❌`，未完成项给出原因与下一步
- **契约影响**：接口、字段、错误码是否变化；若变化说明兼容性
- **偏差说明**：与原 Spec 不一致但已评审通过的调整
- **追踪链接**：关联 PR、Pack、验证记录、回滚信息

可直接使用以下模板：

```markdown
## Changelog
- Date: 2026-02-27
- Status: Done
- Owner: <name>
- Related PR: #123
- Related Pack: `docs/ai/packs/YYYY/MM/<task>.md`

## AC Result
- AC-1: ✅
- AC-2: ✅
- AC-3: ⚠️（延期到 FEAT-456）

## Contract Impact
- API: 无变更
- 字段: 无变更
- 错误码: 无变更

## Notes
- 偏差：<如有，填写评审结论>
- 风险：<剩余风险或后续观察项>
```

## 13) 一句话执行原则

> **先给事实，再给边界，最后要 diff。**  
> **没有证据不落地，越界修改先停下。**



## 14) 问题点
为什么有了spec.md 还要有packs？

- `docs/specs/<feature>.spec.md`：长期规范（相对稳定）
    - 功能目标、业务规则、接口契约、AC
    - 面向产品/架构/评审，生命周期长
- `docs/ai/packs/...`：单次任务执行包（高频变化）
    - 本次 Scope 白名单、运行时报错、UI数值、最小API片段、验证命令、回滚方案
    - 面向开发/AI执行，生命周期短

关系通常是 1:N：

- 一个 `feature.spec.md` 可能拆成很多任务包（按钮样式、埋点、防连点、错误处理…）
- 每个 pack 都引用同一个 spec，但执行细节不同

为什么不能只用 spec？

- spec 不会包含每次调试现场证据（Console/Network）
- spec 不应塞入临时噪音（具体报错、这次只改哪两个文件）
- AI落地最需要的是“这次任务上下文”，这正是 pack 的作用

可以把它理解成：Spec = 蓝图，Pack = 本次施工单。

## 15) 示例

### 15.1 Spec（长期规则） `docs/specs/cart/checkout-button.spec.md`

```markdown
# FEAT-123 购物车结算按钮规范


  ## 2. In Scope
  - 结算按钮交互与样式

  ## 3. Out of Scope
  - 购物车价格计算逻辑
  - 订单创建后端接口协议变更

  ## 4. 业务规则
  - 埋点 payload 必须包含 `cart_id`

  ## 5. 接口契约
  - 请求：`POST /api/v1/checkout`
  - 冲突：返回 `409 Conflict`

## 6. 验收标准（AC）
- AC-1：loading 时按钮禁用，文案为“处理中...”
- AC-2：快速双击只触发一次有效提交

## 7. 风险与回滚
- 风险：防连点逻辑误伤正常点击
```

### 15.2 Pack（本次执行单1：防连点） `docs/ai/packs/2026/02/2026-02-26_FEAT-123_checkout-click-guard.md`

```markdown
- **Spec Ref**: `docs/specs/cart/checkout-button.spec.md`
  - **Goal**: 修复快速双击导致的重复提交，并补齐埋点。
  - **Scope** (files allowed):
    - `src/pages/Cart/SubmitButton.tsx`
    - `src/utils/analytics.ts`
  - **Constraints**:
    - 不修改 public interfaces
    - 不修改全局请求拦截器
    - 仅修改 Scope 文件
  - **UI Facts**:
    - default: bg `#1890FF`, radius `6px`
    - loading: bg `#D9D9D9`, text `处理中...`, `pointer-events: none`
  - **Tech Facts**:
    - `trackEvent(eventName: string, payload: Record<string, any>)`
  - **Runtime Evidence**:
    - 快速双击会发出两次 `/api/v1/checkout`
    - 第二次请求返回 `409 Conflict`
  - **Verification Commands**:
    - 双击结算按钮，确认仅发起 1 次有效请求
    - 检查控制台埋点事件名与 payload 字段
  - **Rollback Plan**:
    - 回滚本次提交并恢复旧按钮逻辑
  - **Output Required**:
    - 先给约束理解摘要
    - 再输出 diff + 回归清单 +风险点
```
